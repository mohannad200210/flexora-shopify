fix: icon font root cause + gallery layout to fill info column height

Two issues, two fixes:

=== ICON FONT REGRESSION — REAL ROOT CAUSE ===

NOT what the user thought (not a missing link, not a different font, not a class name mismatch). The Google Fonts Material Symbols Outlined <link> IS still in the live head and serves a valid CSS with @font-face declarations. Verified.

The real root cause: layout/theme.liquid has this rule in the FLEXORA Arabic styles inline <style> block:

  * {
    font-family: 'Cairo', sans-serif !important;
  }

This universal selector with !important was added in the INITIAL commit (f382bfe), BEFORE the earlier Google Fonts link fix. The * rule with !important always wins over the specific class rule:

  .material-symbols-outlined { font-family: 'Material Symbols Outlined'; }  ← no !important, loses to *

So every <span class="material-symbols-outlined">local_shipping</span> in the page was getting 'Cairo, sans-serif' applied via *, and rendering as text characters in the ligature-font-fallback, not as a Material Symbols icon glyph. The earlier "fix" of adding the <link> was correct (font was loading) but didn't override the !important * rule, so the icons were always broken — the user just didn't visually verify against actual rendered glyphs vs literal text in the first fix.

Why it kept "reappearing" in MORE places: as I built more custom sections (flexora_features, flexora_floating_quote, flexora_trust_lines, the 5 story sections), each new section also used <span class="material-symbols-outlined">icon-name</span>, and every one of them was rendered as literal text by the same * rule. The fix wasn't "overwritten by a later push" — it just never worked to begin with.

Fix: added an explicit !important override on the icon class itself, in the same inline <style> block:

  .material-symbols-outlined {
    font-family: 'Material Symbols Outlined' !important;
  }

Now the icon class wins the !important cascade. Same fix automatically covers ALL sections (collapsible tabs, new custom sections, the icon-bar section, every existing one) because they all use the same .material-symbols-outlined class.

=== IMAGE GALLERY HEIGHT ===

Picked option (b) — add more images to fill the gallery. The theme already supports this through its gallery_layout setting; no new image source code is possible from the theme side. The 16 product images in the live media library are the only images available, so I switched gallery_layout from "thumbnail_slider" (current default — single image + thumbnail strip at bottom, compact) to "stacked" (all images stacked vertically). With 16 images stacked, the gallery column is now ~16x as tall as before and will fill/match the info column's full height.

The sticky behavior (option a) was already enabled by default (enable_sticky_info: true) and was working in the prior push, but it couldn't help here because the gallery was intrinsically short — sticky just keeps a short element in view. With stacked layout, the gallery is now tall enough that sticky genuinely matters.

Also kept media_size: large (default) so each image is full-width and detailed.

No new image assets, no theme-side code beyond the gallery_layout setting change.
